Benvenuto/a nel BlueTeam Plugin!

Questa guida accompagna ogni nuovo operatore SOC nell'attivazione e nell'uso delle 5 skill operative che automatizzano i flussi più frequenti — dalla gestione OOAL alla diagnosi ticket, passando per la checklist mattutina.

Cos'è questo plugin?

Il BlueTeam Plugin è un estensione privata EKO365 per Claude (desktop o CLI) che espone cinque skill orientate al flusso operativo SOC. Ogni skill orchestra in automatico le chiamate al server MCP blueteam, presenta l'evidenza in formato strutturato e — quando necessario — propone la chiusura con un sì/no operatore prima di scrivere qualsiasi cosa.

✅
Sicurezza by design. Le due skill di chiusura (triage-ooal e triage-burst) mostrano sempre una proposta e attendono la tua conferma. Nessuna scrittura avviene senza il tuo Y esplicito.

EKO365 BlueTeam Plugin › Plugin, Skill e MCP

🧩 Plugin, Skill e MCP — come funzionano

Tre concetti fondamentali per capire cosa fa il BlueTeam Plugin sotto il cofano.

🤖

Claude — l'assistente

Claude è il modello linguistico di Anthropic che fa da cervello dell'intero sistema. Nelle sessioni Cowork o Claude Code, Claude non si limita a rispondere: ragiona, pianifica, chiama strumenti esterni e presenta i risultati in formato strutturato per l'operatore.

Nel nostro SOC, Claude legge i ticket su Desk365, consulta Entra Admin Center e SaaS Alerts via MCP, applica le nostre regole di classificazione e — solo su tua conferma — scrive la nota di chiusura.

📦

Plugin — il pacchetto

Un plugin è un pacchetto installabile che estende le capacità di Claude aggiungendo skill, configurazioni e connessioni a strumenti specifici. Si installa una volta sola e resta disponibile in tutte le sessioni.

Il BlueTeam Plugin è un plugin privato EKO365 che raggruppa le 5 skill operative SOC. Una volta installato, Claude sa come comportarsi quando gli chiedi di analizzare un ticket OOAL o avviare la checklist mattutina — perché il plugin gli fornisce le istruzioni.

Analogia: pensa al plugin come alla nostra procedura operativa standard caricata in memoria. Invece di aprire manualmente Desk365, Entra Admin Center e il portale SaaS Alerts seguendo un protocollo, Claude lo fa in automatico — rispettando esattamente le stesse regole operative del nostro SOC.
⚡

Skill — il flusso operativo

Una skill è un flusso operativo pre-configurato che Claude esegue quando viene invocato con il comando slash corrispondente. Ogni skill definisce: quali dati raccogliere, in quale ordine, come interpretarli e come presentare il risultato all'operatore.

Nel BlueTeam Plugin, ogni skill corrisponde a un tipo di compito SOC ricorrente. Invece di aprire manualmente Desk365, Entra Admin Center e il portale SaaS Alerts uno per uno — come facevamo prima — scrivi un comando e Claude fa tutto il lavoro di raccolta. Tu validi e decidi.

ComandoCosa fa la skill
/blueteam:morning-checklistAggrega coda, salute agente, backlog
/blueteam:triage-ooal <N>Analizza e propone chiusura ticket OOAL
/blueteam:triage-burst <N>Gestisce burst File Limit (padre + figli)
/blueteam:investigate-user <email>Profila un utente M365 completamente
/blueteam:diagnose-stale <N>Diagnostica perché un ticket è fermo
🔌

MCP Server — il ponte verso i sistemi

MCP (Model Context Protocol) è il protocollo standard che permette a Claude di interagire con i nostri sistemi in modo controllato e sicuro. Il server MCP blueteam è il componente che espone a Claude le operazioni sui nostri portali SOC.

Senza il server MCP, Claude potrebbe ragionare ma non aprire Desk365 per leggere i ticket, non consultare Entra Admin Center per i log di accesso, non scrivere la nota di chiusura. È il server MCP che abilita quelle operazioni — con guardrail integrati (idempotenza, autorizzazioni, audit log).

Strumenti esposti dal server MCP blueteam
scan_queue — coda ticket su Desk365
ticket_details — dettaglio ticket su Desk365
user_security_overview — profilo da Entra Admin Center
signin_logs — Sign-in logs su Entra Admin Center
device_check — device dell'utente su Intune
argentino_health — stato agente autonomo
ticket_diagnose — diagnosi da Desk365 + SaaS Alerts
close_ticket — scrive la chiusura su Desk365 ✱
✱ Unico strumento in scrittura — richiede conferma operatore e ha guardrail server-side.

⚙️ Installazione del plugin

Segui questi passaggi una sola volta. Tempo stimato: ~5 minuti.

⚠️
Prerequisiti: Claude desktop (o Claude Code CLI) installato e funzionante, GitHub CLI (gh) autenticato con accesso a eko365sb. Le credenziali per Desk365/Graph/SaaS Alerts vengono configurate nel prossimo capitolo (🔐 Setup credenziali) via Bitwarden Vault. Non è più necessario nessun .env manuale.

1
Aggiungi il marketplace EKO365 (una sola volta)

claude plugin marketplace add eko365sb/claude-plugins
2
Installa il plugin BlueTeam
Scarica e attiva il plugin dalla sorgente EKO365.
claude plugin install blueteam@eko365-plugins
3
Verifica l'installazione
Apri una nuova sessione Claude e lancia la checklist mattutina. Se vedi l'output strutturato, il plugin è attivo.

EKO365 BlueTeam Plugin › Setup credenziali

🔐 Setup credenziali (Bitwarden Vault)

Una sola volta al primo setup. Tempo stimato: ~3 minuti.

🔑
Le tue credenziali sono in Bitwarden Vault. L'MCP BlueTeam le legge automaticamente al boot. Tu non vedi mai i valori — vedi solo il risultato delle query (ticket, alert, signin, ecc.).

Il flusso di setup

FABIO (admin) genera Access Token in Bitwarden web UI
│
▼ Bitwarden Send (link cifrato monouso)
TU (operatore) ricevi token + Organization ID
│
▼ eko365-blueteam-init --token US:... --org ...
~/.eko365/bitwarden_config.env creato sul tuo PC
│
▼ source ~/.eko365/bitwarden_config.env
MCP boot reads from vault
│
/blueteam:morning-checklist OK

Procedura

1
Ricevi Access Token e Organization ID dall'admin
L'admin (Fabio) ti invia 2 stringhe:
  • Un Access Token (formato 0.xxx...) tramite Bitwarden Send (link cifrato monouso)
  • L'Organization ID (UUID), tipicamente f090dcd7-df9e-43db-b230-b440011f75c5 per EKO365
⚠️
Non condividere il token. Chi lo possiede può leggere tutte le credenziali EKO365 BlueTeam.
2
Esegui il comando di init
Sul tuo terminale (PowerShell su Windows, Terminal su Mac), nella directory del workspace BlueTeam:
./scripts/eko365-blueteam-init.py \
  --token "0.xxx-paste-the-real-token-here-xxx" \
  --org "f090dcd7-df9e-43db-b230-b440011f75c5"

3
Attiva il vault nella tua shell

echo '[ -f ~/.eko365/bitwarden_config.env ] && source ~/.eko365/bitwarden_config.env' >> ~/.zshrc

Problemi comuni

SintomoCausaCosa fare
Init dice "Token format invalid" Hai incollato il token con caratteri extra (spazi, virgolette, ecc.) Re-incolla pulito; deve iniziare con 0. e essere lungo >30 caratteri
Init dice "Authentication failed" Token revocato o expired Chiedi un nuovo Access Token all'admin (Fabio)
Doctor dice "Status: DEGRADED" con vault non disponibile Hai dimenticato source ~/.eko365/bitwarden_config.env, oppure è scaduto il token Source il file e re-run doctor; se ancora DEGRADED, chiedi nuovo token
MCP parte ma morning-checklist dà errore 401 Un secret nel vault è stato ruotato e il valore upstream è cambiato Restart Claude Code (per forzare reload cache); se persiste, contatta admin

Se devi reinstallare

Laptop nuovo, format, ecc.:

  1. Chiedi all'admin un nuovo Access Token (il vecchio andrebbe revocato per pulizia)
  2. Reinstalla Claude Code + plugin BlueTeam come da capitolo Installazione
  3. Esegui di nuovo eko365-blueteam-init.py con il nuovo token

Sicurezza e responsabilità

🔒
Cosa NON fare:
  • Non copiare ~/.eko365/bitwarden_config.env su altri dispositivi
  • Non condividere screenshot del terminale che mostrano il token (anche solo visivamente)
  • Non lasciare il PC sbloccato in luoghi pubblici
Se sospetti un problema (laptop rubato, malware, accesso non autorizzato): notifica l'admin entro 1 ora. La revoca del tuo Access Token richiede meno di 5 minuti e blocca completamente l'accesso al vault.

Skills › morning-checklist
🌅

morning-checklist

Dashboard sintetica giornaliera per l'operatore SOC. Aggrega coda corrente, salute di Argentino, ticket assegnati e backlog stagnante in un'unica lettura decision-ready.

Sola lettura 1-2 volte al giorno morning checklist riassunto giornata status BlueTeam

Come si usa

/blueteam:morning-checklist 📋

Nessun argomento richiesto. Lancia all'inizio del turno, o in qualsiasi momento per aggiornare il quadro.

Puoi anche dirlo così — Claude capisce e avvia la skill
Com'è la coda stamattina? Ci sono ticket fermi su Kryalos o VBC?
Dammi un quadro della situazione prima di iniziare il turno.
Come sta Argentino? Ha lavorato bene stanotte?
Ci sono ticket assegnati a me su Desk365?

Cosa fa internamente

  • 1
    Scansiona la coda (scan_queue)
    Recupera tutti i ticket aperti su tutti i tenant. Distingue quelli in lavorazione umana (Pending / In progress + assegnatario) da quelli disponibili per il triage.
  • 2
    Verifica salute Argentino (argentino_health)
    Controlla liveness, ultimo run, conteggio final-fail 48h, fast-path attivati. Classifica lo stato in GREEN / YELLOW / RED.
  • 3
    Ticket assegnati (assigned_tickets)
    Aggrega per operatore i ticket in carico. Utile per vedere cosa sta facendo l'altro operatore senza dover aprire Desk365.
  • 4
    Backlog stagnante
    Estrae dalla risposta di argentino_health i ticket aperti da oltre 24 ore con assegnatario e soggetto.

Output atteso

Triage #15634 (OOAL stagnante, candidato R17d) 2. Analisi #15621 (burst download, verificare padre/figli) 3. Coda rimanente per ordine di arrivo
⚠️
Attenzione: se Argentino mostra external_error, non allarmarti subito. Conta i final-fail, non i tentativi. Se i final-fail sono 0 e lo stato è recovered, l'agente funziona correttamente.
Skills › triage-ooal
📍

triage-ooal

Flusso completo per ticket Outside Approved Location (OOAL). Recupera l'evidenza autoritativa, classifica con regole rapide, attiva il ragionamento LLM se necessario, propone chiusura e scrive su Desk365 solo dopo la tua conferma.

Lettura + chiusura ~15 / settimana triage OOAL outside approved triage location

Come si usa

/blueteam:triage-ooal <numero-ticket> 📍

Se dimentichi il numero, la skill te lo chiede.

Puoi anche dirlo così — Claude capisce e avvia la skill
Kryalos
mario.rossi@kryalos.com ha un accesso da Praga nel ticket #15712 — è pulito?
VBC
Ho un outside approved location su VBC che non torna — guarda il #15698.
DLPHRC
anna.bianchi@dlphrc.it si è collegata dalla Rep. Ceca, è nel ticket #15701. Lo escaliamo?
C2Partners
Ticket OOAL #15689 su C2Partners — fammi vedere l'evidenza e proponi.

Flusso operativo (8 passi)

  • 1
    Recupero contesto base
    Legge il ticket da Desk365. Se è già chiuso, o assegnato ad altri, si ferma e te lo segnala prima di procedere.
  • 2
    Estrazione metadata
    Dal soggetto del ticket estrae tenant_code e email utente (es. C2PARTNERS → C2P, DLPHRC → DLP).
  • 3
    Profilo utente (168h)
    Recupera sign-in Graph, device Intune, eventi SaaS Alerts delle ultime 7 giorni.
  • 4
    Source event autoritativo
    I ticket Respond rule hanno IP "Not Provided" nel corpo. La skill recupera automaticamente l'evento sorgente outside.own.location su SaaS Alerts — quello con l'IP reale.
  • 5
    Classificazione con regole rapide
    Tenta le regole in ordine di priorità. Alla prima corrispondenza, propone la chiusura.
  • 6
    LLM Reasoning (se nessuna regola corrisponde)
    Analisi guidata con evidenza sintetica, ragionamento passo-passo e confidenza ALTA/MEDIA/BASSA. Se BASSA → nessuna proposta, escalation obbligatoria.
  • 7
    Decision point
    Mostra evidenza, verdict e note proposte. Aspetta il tuo Y / n / explain.
  • 8
    Esecuzione
    Su Y chiude il ticket via close_ticket. Su n registra il rifiuto senza scrivere nulla. Su explain apre un dialogo per il verdict alternativo.

Le regole rapide (gerarchia)

R12
Grace period: OOAL precedenti recenti dalla stessa location negli ultimi 7 giorni. Viene valutata per prima come ottimizzazione — se corrisponde, non serve raccogliere il resto dell'evidenza.
R17
Italia, device compliant, tutto pulito: Country IT + device Intune compliant + sign-in success + risk none + CA success + ESCALATE_IDENTITY. Caso tipico: VPN aziendale.
R17b
Italia, backbone ASN: Country IT + ASN backbone (Level3, Cloudflare, HE, GTT, NTT...) + device compliant. IP backbone = geolocalizzazione inconsistente, false positive frequente.
R17c
Italia, backbone, device personale: Come R17b ma device non-compliant (BYOD). Richiede baseline same-IP ≥ 2 nelle ultime 4 settimane come ulteriore conferma.
R17d
EU/OECD, non datacenter: Country EU o OECD (CZ, FR, DE, GB, US...) + ASN non datacenter + device compliant + baseline ≥ 2 + threat clean. Tipico caso trasferta.

Output — esempio decision point

Device aziendale compliant. Baseline 3 accessi stesso IP 30gg. Chiusura automatica R17d. [BlueTeam Fast-Path]" ▶️ Procedi con close FP_CERTO (15 min)? [Y/n/explain]
💡
Quando rispondere "explain": scrivi explain se vuoi capire il ragionamento dietro la proposta prima di decidere. La skill entra in modalità interattiva e risponde alle tue domande.

triage-burst

Flusso operativo per ticket File Download/Upload Limit Exceeded. Identifica automaticamente il parent e tutti i children del burst, applica le regole R16, chiude il set in un'unica operazione.

Lettura + chiusura ~10 / settimana triage burst file download limit bulk download bulk upload

Come si usa

/blueteam:triage-burst <numero-ticket> 📂

Puoi passare uno qualsiasi dei ticket del burst — la skill individua da sola il ticket padre e tutti i figli.

Puoi anche dirlo così — Claude capisce e avvia la skill
SACMAR
anna.verdi@sacmar.it ha generato 4 ticket File Download Limit in 10 minuti. Gestisci il burst dal #15700.
VBC
marco.bianchi@vbc.it — bulk upload OneDrive, 3 ticket aperti. Il padre dovrebbe essere il #15702.
C2Partners
Ticket #15698 File Limit su C2Partners — controlla se ci sono fratelli e chiudi tutto.
Kryalos
C'è un burst di download su Kryalos. Ho visto il #15710 ma potrebbero essercene altri.

Il concetto di burst: ticket padre e ticket figli

💡
Un burst File Limit genera più ticket per lo stesso utente e stesso tipo di evento in una finestra di ~1 ora. Il ticket padre è il più vecchio (base della fatturazione: 15 min). I ticket figli sono i successivi (0 min — il costo è già sul padre).

Flusso operativo

  • 1
    Recupera contesto base + verifica idempotenza
    Legge il ticket passato. Controlla se già chiuso, se assegnato ad altri.
  • 2
    Identifica il burst
    Estrae email utente, tipo evento (download/upload), finestra temporale. Poi scansiona la coda del tenant cercando ticket fratelli dello stesso utente nella finestra ±1h.
  • 3
    Determina il parent
    Il ticket con created_on più vecchio è il parent. Tutti gli altri sono children.
  • 4
    Classifica il parent (regole R16)
    Applica R16a (folder-zip download), R16d (bulk upload OneDrive sync), R16b (recurrent downloader storico ≥3/90gg). Se nessuna corrisponde, attiva LLM reasoning.
  • 5
    Decision point + esecuzione
    Mostra il set completo (parent + children), verdict e billing. Su Y chiude prima il parent (15 min), poi ogni child (0 min) in sequenza. In caso di errore su un child si ferma — gli altri restano aperti.

Le regole R16

R16a
Folder-zip download: Joint type download + User-Agent ms_internal (es. ODMTADemand-Transform_Zip) + hosting Microsoft Corporation + threat clean. Pattern tipico: ZIP di cartella SharePoint.
R16d
Bulk upload OneDrive sync: Joint type upload + ASN Microsoft Corporation hosting + target = drive personale utente + threat clean. Pattern tipico: sincronizzazione OneDrive su nuovo device.
R16b

EKO365 BlueTeam Plugin › Anti-pattern

🚫 Anti-pattern operativi

Cosa non fare mai — raccolta di errori comuni e comportamenti da evitare.

Triage OOAL

✗
Non affidarsi al solo country mostrato in Entra Admin Center per IP backbone. Level3, Lumen, Cloudflare, Hurricane Electric, GTT, NTT: la geolocalizzazione è inconsistente tra la GeoDB Microsoft e MaxMind/IPinfo. Il source event nel portale SaaS Alerts è il dato autoritativo per il paese di accesso.
✗
Non applicare R17 a country diverso da IT. R17 è solo per Italia. Per paesi EU/OECD usa R17d.
✗
Non chiudere se baseline 30gg = 0 e country non in approved list. Se l'utente non ha mai avuto accessi da quella location, meglio escalation manuale.
✗
Non fidarti di trust_score < 80 anche se gli altri segnali sono positivi. Declassare a confidenza MEDIA o BASSA.
✗
Non inventare pattern simili a regole rapide se non corrispondono esatto. Meglio LLM reasoning con disclaimer chiaro che forzare una regola rapida non calzante.

Triage burst

✗
Non chiudere i children con minuti > 0. Il billing è aggregato sul parent per design. I children vanno sempre a 0 min. (vedi commit 0a014b8 per il bug storico sistemato)
✗
Non applicare R16a/R16d se is_datacenter=True ma l'ASN non è Microsoft. AWS, GCP o altri datacenter generici potrebbero essere exfiltration. Escalation manuale.
✗
Per upload bulk: verificare che il target sia il drive personale dell'utente. Se è un shared link verso esterni → escalation. Non è un caso OneDrive sync normale.

investigate-user

✗
Non usare investigate-user per concludere FP o escalation. È solo profilazione. L'output è input per le skill di analisi ticket o per una valutazione manuale.
✗
Se Argentino è in stato di degrado dei provider, dichiarare l'evidenza parziale. Alcuni dati potrebbero mancare — non trattare l'assenza di segnali negativi come conferma di sicurezza.

diagnose-stale

✗
Non proporre chiusura del ticket da diagnose-stale. Questa skill è solo diagnostica. La chiusura avviene con triage-ooal o triage-burst.
✗
Non inventare cause se i dati non sono chiari. Dichiarare "cause incerte, serve verifica manuale" è la risposta corretta quando i dati sono ambigui.

morning-checklist

✗
Non allarmarsi per Argentino in external_error se i final-fail sono 0. Conta i final-fail, non i tentativi. Errori esterni recovered non indicano un problema reale.

Generale

✗
Non sovrascrivere ticket con cf_Triage Source = "AI-Agent". Significa che Argentino ha già processato il ticket. Non sovrascrivere senza aver capito perché è rimasto aperto.
✗
Non procedere senza conferma per le skill di chiusura. Rispondere Y in modo riflessivo senza leggere l'evidenza vanifica il valore del sistema. Il punto è che tu validi.
